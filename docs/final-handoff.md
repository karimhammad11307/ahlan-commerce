## Mentor Defense Preparation

**1. Can you explain the system from request to database?**
A client request hits our deployed Nginx or Axum server. Axum's routing layer receives the HTTP request and runs it through middleware (tracing/CORS). The handler unpacks the request, passes it to the Service layer for business logic, which in turn calls the Data Access Layer (DAL) generated by Cornucopia. The DAL securely executes raw SQL against the PostgreSQL database.

**2. Can you explain domain ownership?**
The system is divided into logical boundaries (e.g., Products, Users, Orders). The "domain" owns its specific entities and business logic. External domains cannot directly query the database tables of another domain; they must use the exposed internal service methods or public APIs to ensure data integrity and encapsulation.

**3. Can you explain route/config constants?**
Hardcoded strings (like route paths or config keys) are centralized as constants. This prevents typo-driven bugs, allows the compiler to enforce correctness, and makes future refactoring of endpoints or env-var names simple and globally consistent.

**4. Can you explain the public error envelope and private root-cause context?**
When an error occurs, the Rust backend logs the exact line number, stack trace, and database failure reason (the "root-cause") to the server terminal. However, it returns a sanitized, vague JSON response (the "public envelope") to the user (e.g., `{"error": "Internal Server Error"}`). This prevents leaking sensitive schema details or database states to malicious actors.

**5. Can you explain what request tracing fields are logged and why?**
We log the HTTP method, URI, request ID, duration, and status code. We log these to provide observability. If a specific request fails or is slow, we can use the unique request ID to trace the exact sequence of events in the logs without having to guess which user triggered it.

**6. Can you explain where UUIDv7 IDs and timestamps are created?**
They are created in the Rust application layer (the backend service) rather than relying on PostgreSQL `DEFAULT gen_random_uuid()` or `NOW()`. This allows the backend to know the exact ID and timestamp of an entity before it is even inserted, making integration testing, caching, and related insertions much easier.

**7. Can you explain the Cornucopia-generated DAL?**
Cornucopia is a tool that takes our raw SQL queries (`.sql` files), validates them against a live database schema, and generates type-safe Rust structs and functions. This means if we write a bad SQL query or if the schema changes, the Rust code simply will not compile, eliminating runtime SQL errors.

**8. Can you explain Redis cache keys and invalidation?**
We use Redis to store expensive API responses. Cache keys are deterministic (e.g., `product:detail:UUID`). When a product is updated or deleted, the service layer explicitly sends a command to Redis to delete (invalidate) that specific key, ensuring the next GET request fetches fresh data from PostgreSQL.

**9. Can you explain worker recovery?**
The background worker continually polls the database or message queue for jobs. If the worker crashes mid-job, the job remains in a "pending" or "processing" state. Upon restart, the worker queries for abandoned jobs (e.g., processing for > 10 minutes) and retries them, ensuring no background task is lost due to a crash.

**10. Can you explain generated OpenAPI docs vs written docs?**
OpenAPI docs are automatically generated from our Rust codebase. They guarantee that the API documentation is always 100% accurate and mathematically aligned with the code. Written docs (like `README.md` or `deployment.md`) are for human context—explaining *why* the system exists and *how* to deploy it, which code cannot explain.

**11. Can you explain how the deployed API and worker start?**
Both use the exact same compiled Docker image (`ahlan-api`). The API starts using the default command `./api`, which binds to `0.0.0.0:3000` and listens for web traffic. The worker starts by overriding the Docker command with `./api worker`, which skips starting a web server and immediately drops into a polling loop for background jobs.

**12. Can you explain what CI proves before Coolify deploys?**
CI proves that the code formats correctly, passes all strict linters (`clippy`), passes all unit and integration tests, and that the Cornucopia SQL bindings perfectly match the database schema. If CI is green, we have mathematical confidence the app will boot and function in Coolify.

**13. Can you explain why PRD and ADR exist?**
The PRD (Product Requirements Document) defines *what* we are building and *why* it matters to the user. The ADR (Architecture Decision Record) defines *how* we decided to build it technically and *why* we didn't choose alternatives. They protect the team from building the wrong feature or endlessly debating solved technical decisions.

**14. Can you explain every AI-assisted diff?**
Any code written by an AI agent was explicitly reviewed by a human engineer against the project's constitution, architecture notes, and guard-skills (like Clean Code and Security guards). No code is committed blindly; the human engineer maintains ultimate ownership and understanding of the exact logic applied.

**15. Can you debug the Coolify deployment from logs and docs?**
Yes. By referencing `docs/deployment.md`, we know exactly what environment variables the server expects. If the server crashes, we can look at the Coolify GUI logs, find the exact `tracing` root-cause error, and immediately isolate whether it's a missing `DATABASE_URL`, a failed migration, or a code panic.
